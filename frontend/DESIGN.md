# タイプクエスト 全体設計ドキュメント

「なぜこの形にしたか」を残すための設計ドキュメント。ストーリーモードの詳細設計は
[`app/story/DESIGN.md`](app/story/DESIGN.md) に分けてあるので、そちらを参照。
このファイルはアプリ全体（ルーティング・認証・試練の塔・共通バトルエンジン）を扱う。

新しいシステムを足すときは、まずここ（または該当する `DESIGN.md`）に設計を書き足してから
実装する運用にしたい。

## 1. 概要

Next.js App Router（v16.2.9、Tailwind CSS v4、TypeScript）で作る、日本語タイピングRPG。
現状はフロントエンドのみでバックエンドが無い（3節参照）が、Goで書いたバックエンドを
今後追加する予定。実装するときは `backend/DESIGN.md` にAPI設計・DB設計をまとめてから
着手すること（現時点ではまだ書かれていない）。

`frontend/AGENTS.md` に「このNext.jsは訓練データと違う破壊的変更が入っている前提で
`node_modules/next/dist/docs/` を見てから書け」という注意書きがあるので、Next.js固有の
APIを使う前は一度そこを確認すること。

## 3. 認証・ログイン（現状は本物のバックエンドが無い簡易実装）

`backend/` フォルダは存在するが中身が空。API Routeも一切無く、外部サーバーへのfetchも無い。
つまり**認証はローカルの見せかけ**：

- 登録（`app/register/page.tsx`）：`userName`/`email`/`password` を平文のまま
  `localStorage` に直接書き込むだけ。
- ログイン（`app/login/page.tsx`）：`localStorage` に保存済みの email/password と
  入力値を単純な文字列比較し、一致したら `localStorage.setItem("isLoggedIn", "true")`。
- ホーム画面（4節）はこの `isLoggedIn` フラグだけを見てロック解除を判定する。

ハッシュ化・セッション・DB・複数ユーザー対応は無い（同じブラウザで登録し直すと
前の資格情報を上書きする、決め打ちキーのシングルユーザー状態）。1節のGoバックエンドを
作るタイミングで、ここを本物の認証に設計し直す予定。

## 4. ホーム画面とモード選択のゲート（`app/home/page.tsx`）

3つのモードカード：
- 旅に出る（ストーリー） → `/story`、`locked: true`
- 闘技場（オンラインモード表記） → `/battle`、`locked: true`。実体はオンライン対戦
  要素が無く、共通バトルエンジンにそのまま入るだけ（`mode`未指定＝`practice`扱い相当）
- 試練の塔 → `/practice`、`locked: false`（誰でも入れる）

加えて、右上に設定アイコン → `/setting` を置く予定（2節参照。誰でも入れるが、中身は
ログイン状態によって表示が変わる）。

`locked: true` のカードを選んだ状態でクリック/Enterすると、`isLoggedIn` でなければ
ログイン誘導のダイアログを出す。矢印キーでのカード選択、クリックでの選択の両方に
同じロックチェックが重複して書かれている。

## 5. 試練の塔（`app/practice/`）

- `page.tsx`：初級/中級/上級の3枚のカードから `/battle?mode=practice&level=easy|normal|hard`
  へ遷移するだけの薄い画面。初回訪問時は自動でチュートリアルダイアログが開く
  （`showTutorial` 初期値 `true`）。「あそびかた」ボタンでいつでも再度開ける。
  チュートリアル表示中は矢印キーでのカード選択を無効化している。
- `tutorial.tsx`：5枚組のカルーセル（embla-carousel）。①白/黒の単語色＝自分の攻撃/
  敵の攻撃の区別、②敵を倒すと残り時間が増える、③1/2/3キーで炎/氷/雷の魔法を選択
  （再押しで解除）、④階層が上がるほど単語が長く・速くなり5階ごとにボス、⑤階層ごとに
  宝箱でアイテムが手に入り、Enterを押すまでゲームが一時停止する、という流れを説明。
  **修正済みのバグ**：ダイアログに `max-w-2xl` を指定していたが、ベースの
  `DialogContent`（`components/ui/dialog.tsx`）が `sm:max-w-sm` を持っており、
  Tailwindが生成するCSSではレスポンシブ指定（`sm:`）が無指定のクラスより後ろに
  並ぶため、640px以上の画面では `sm:max-w-sm` が後勝ちしてダイアログが384pxまで
  縮み、カルーセルや文章が横にあふれていた。`className`側を `sm:max-w-2xl`
  （同じ`sm:`バリアントで指定）に直すことで、`tailwind-merge`が正しく後勝ちさせて
  くれるようになり解決した。同じ理由で他のダイアログの幅を広げたいときも、
  ベースのブレークポイント（`sm:`）に合わせて指定すること。
- `/battle` への引き継ぎは `mode`/`level` の2つのクエリだけ（ストーリーモードは
  これに加えて `hp`/`maxHp`/`words`/`attackBooks`/`defenseBooks`/`encounter` 等
  多数を渡す。試練の塔とストーリーで必要な情報量がそもそも違うことの表れ）。

## 6. 共通バトルエンジン（`app/battle/`）

試練の塔・ストーリーモードの両方が同じ `battle/page.tsx` を共有し、`mode`
（`"practice"` | `"story"`）で随所を分岐する（`isStory` フラグ）。ストーリー
モード固有の部分（HP持ち越し、単語プールの分離、コトの復活演出、言葉ドロップなど）は
[`app/story/DESIGN.md`](app/story/DESIGN.md) の6〜7節にまとめてあるので、
ここでは**試練の塔側の一般的な仕組み**を書く。

### 6.1 階層進行
`floor` は1から始まり、フロアの敵を全滅させると `grantTreasure()`（6.2節）を呼んで
一時停止し、Enterで次の階層へ。5階ごと（`isBossFloor`）はボス階になり、`enemyList.boss`
からランダムで1体だけ出現、HP/スコア/攻撃力に`BOSS_HP_MULTIPLIER`(2)/
`BOSS_SCORE_MULTIPLIER`(4)/`BOSS_ATTACK_MULTIPLIER`(1.2)がかかる。ボス階以外は
`getEnemyCountForFloor`（3階ごとに1体増、最大3体）、`getEnemyAttackDamage`
（1階につき+8%）、`getWordSpeedForFloor`（1階につき+5%）でじわじわ強くなる。
出てくる単語の長さ分布も `pickWordForFloor` が階層に応じてガウス重み付けで
長い単語のほうへシフトさせる（5段階の長さティア。ストーリーモードもこの関数を
流用している＝6節の設計思想が全体で共有されている）。

### 6.2 宝箱・アイテム
階クリアごとに6種類（`attack`/`defense`/`hp`/`fireMagic`/`iceMagic`/`lightningMagic`）
から1つランダムで付与（`grantTreasure`）。attack/defenseは倍率を`ITEM_ATTACK_BONUS`/
`ITEM_DEFENSE_BONUS`(共に0.1)ずつ強化（defenseは`ITEM_DEFENSE_MULTIPLIER_MIN`(0.3)で
下限あり）、hpは`ITEM_HP_BONUS`(20)ぶんさいだいHPを上げて全回復、魔法系はその魔法の
チャージを1つ増やす。Enterが押されるまでゲーム（単語のスクロール・残り時間）を
完全に一時停止する。

### 6.3 魔法システム
敵種族ごとに固定の`dropsMagic`があり、撃破時に`MAGIC_DROP_CHANCE`(0.35)の確率で
その魔法を1チャージ入手（ボスは確定ドロップ）。自分の攻撃フェーズ中のみ1/2/3キーで
発動する魔法を選択（再度押すと解除）。炎＝ダメージ`FIRE_DAMAGE_MULTIPLIER`(2倍)、
氷＝次の3単語ぶん流れる速度を`ICE_SLOW_SPEED_MULTIPLIER`(0.4倍)に、雷＝生存中の
敵全員に同時ダメージ（`resolveLightningAttack`）。魔法使用時は会心が出ない。

**今後の予定**：光魔法・回復魔法などの追加。追加する際は`MagicType`・`MAGIC_META`・
`ITEM_TYPES`（宝箱ドロップ対象）・チュートリアルの説明文の4箇所を揃えて直すこと。

### 6.4 コンボ・会心・ミスペナルティ
単語を打ち切るたびにコンボが+1、時間切れでリセット。会心率`getCritChance`は
コンボ×0.03（上限50%）、会心ダメージは`CRIT_DAMAGE_MULTIPLIER`(1.5倍)。ミスは
その「ターン」内での回数に応じて`getMissPenaltyMultiplier`でダメージ/防御効果を
弱める（階層が上がるほど1ミスの重みが増す、最低でも1割は残る）。単語自体の
「威力」は`getWordPower(word) = length * (length + 5)`（文字数の2乗に比例させて、
短い単語の連打が長い単語より強くならないようにしてある。この式の設計理由は
`battle/difficulty.ts`内のコメントに詳しい）。

### 6.5 ローマ字入力エンジン（`romaji.ts`）
`ROMAJI_TABLE`が各かな（拗音は2文字セットも）を許容するローマ字表記の配列に
マッピングしている（先頭が既定表示）。`buildWordUnits`がかな単語を「打鍵単位」に
分解する際の工夫：
- 拗音（きゃ等）は1つの単位として消費
- 促音（っ）は単独の打鍵にせず、次の単位の子音を重ねる形にたたみ込む
  （「った」→ `["tta"]`）
- 「ん」は**ロールオーバー処理**が必要な特殊ケース：単語末尾なら`["n"]`だけで確定
  できる（後に続く文字が無く曖昧さが無いため）。次の単位がa/i/u/e/o/n/yで始まる
  （＝「な」なのか「んな」の続きなのか曖昧になりうる）場合は`["nn"]`のみを受け付け
  二重入力を強制する。それ以外は`["n","nn"]`両方を許容する。
  （このロールオーバー処理は、たんぽぽ等の実際の単語で過去にバグが出た経緯があり、
  ストーリーモードの単語収集で似た問題が再発しないよう、単語追加時は打鍵の
  ロールオーバーが正しく効くか実際に打って確認するとよい）

### 6.6 単語リスト（`words.ts`）
難易度ごとに `{kana, kanji}` の配列。短い単語〜長い単語まで幅広く含み、
`pickWordForFloor` の長さティア抽選で使われる。ストーリーモード専用の単語は
別ファイル（`app/story/storyWords.ts`）に分離してあり、こちらとは共有しない
（`app/story/DESIGN.md` 6節参照）。

### 6.7 時間制限（TODO: 詳細未確定）
単語を打つたびに残り時間が1秒増える（`TIME_BONUS_PER_KILL`とは別の、打鍵そのものへの
ボーナスを検討中）。どの関数・どんな条件で加算するかはまだ決めていない。設計が固まったら
ここに書くこと。

## 7. ストーリーモード

→ [`app/story/DESIGN.md`](app/story/DESIGN.md) を参照。マップ・当たり判定の自動計算、
会話・立ち絵システム、単語収集と難易度カーブ、アイテム経済、メニューUI、セーブ演出など
第1章の設計をまとめてある。

**今後の大きな方針転換（未着手）**：章ごとに独立したマップを作るのをやめて、村・森などを
含んだ1枚の全体マップを歩き回るRPGにしたい。ボスを倒すごとに次の村へ行けるようになり、
それまではボスの周辺に強力な敵を配置したり、進もうとすると「村を〇〇（例：スライムキング）が
襲っている。助けに行こう」のように呼び止められて先に進めない、という形の進行ゲートにする
（第1章の「必須の言葉を3つ集めるまで村を出られない」というゲート――9節参照――は、この
大きな仕組みの先行実装にあたる）。マップ構造そのものの作り直しが必要な大きめの変更なので、
着手する際はまずこのファイルに具体的な設計（マップの繋がり方、章の境界をどう表現するか等）を
書き足してから始めること。

## 8. 共通UIキット（`components/ui/8bit/`）

`badge` / `button` / `card` / `carousel` / `dialog` / `input` / `progress` の
8bit風（ドット絵アニメ）ラッパーコンポーネント群。shadcn/ui＋Radix UIの上に
レトロゲーム風の見た目（`DotGothic16`フォント、太い枠線）を被せてある。
`components/HomeButton.tsx` はどの画面からでも `/home` に戻れる共通ボタン。

## 9. 技術スタック

Next.js 16.2.9（App Router）／React 19.2.4／TypeScript 5／Tailwind CSS v4／
Radix UI／class-variance-authority・clsx・tailwind-merge（`cn()`ユーティリティ）／
embla-carousel-react（チュートリアルのカルーセル）。HTTPクライアントやDB・認証系の
パッケージは一切入っていない（3節の通り、バックエンドが無いことと整合している。
1節のGoバックエンドが実装されたら、フロントエンドからのHTTPクライアント追加も
このセクションに書き足すこと）。
