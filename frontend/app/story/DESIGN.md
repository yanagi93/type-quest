# ストーリーモード 設計ドキュメント（第1章「はじまりの草原」）

このドキュメントは、`frontend/app/story/` 以下に実装済みのストーリーモード第1章の設計をまとめたもの。
コードを読めば「何をしているか」はわかるが、「なぜこの形にしたか」はここに書く。
新しいシステムを足すときは、まずここに設計を書き足してから実装する運用にしたい。

## 1. 概要

タイピングRPG「タイプクエスト」の中で、ストーリー仕立てで探索・単語収集・戦闘を通しで遊べるモード。
試練の塔（`frontend/app/practice/`, `frontend/app/battle/` の非ストーリー部分）とは独立した仕組みで、
`frontend/app/battle/page.tsx` は `mode=story` のクエリパラメータで分岐して両方を共有している。

ゲームループ：
```
村を探索 → NPC/物にぶつかって言葉を覚える → 門から外（世界地図）へ出る
→ 歩くとランダムで戦闘（typing）→ 一定数の必須言葉を覚えたらボスに挑戦できる
→ ボスを倒すと「ぼうけん」を覚えて次の章へ
```

## 2. シーン構成（ステートマシン）

`useStoryState.ts` の `StoryScene`：

```
select → prologue → intro（夢） → elderVisit（長老と本を渡される） → meetKoto（コトと出会う）
→ town ⇄ field（探索、GridExplorerで自由移動）
→ battle（別ページ /battle。往復はlocalStorage経由）
```

- `select`（章選択画面）は廃止し、`TitleScreen.tsx`（「はじめから」/「つづきから」）に
  置き換えた。「つづきから」はオートセーブ＋手動セーブ3枠から選んで再開する
  （`SaveSlots.tsx`、`useStoryState.ts`のセーブ関連関数を参照）
- `prologue`〜`meetKoto` はいずれも `StoryDialogue` を使った一本道の会話シーン（詳細は4節）。
- `town`＝村の中（詳細な1:1スケールのマップ、`TOWN_MAP`）、`field`＝村の外に広がる
  世界地図（村より大きな縮尺のミニマップ、`FIELD_MAP`）。それぞれ別の `GridExplorer` マップで、
  村の門にぶつかるとシーンごと切り替わる（詳細は4.2節）。
  **経緯**：一時期、章ごとにマップを切り替えるのをやめて村と草原を1枚の地続きマップに
  統合していたが、「本当にドラクエのように、城や町から出たら世界を冒険しているような
  マップで縮図にしてほしい」という要望を受けて、村（詳細マップ）と世界地図（縮図）を
  別マップに戻した。ドラクエで町に入ると町専用の詳細マップに切り替わるのと同じ考え方。
- 章の途中でも「📖 章選択に戻る」でいつでも `select` に戻れる（確認なし）。←なくしてください。セーブしたデータに戻ることは可能

### なぜ `intro` を2つに割ったか（`intro` → `elderVisit`）
最初は「夢を見る→長老に本を渡される」を1つの `StoryDialogue` にまとめていたが、
長老が直接セリフで喋ると、おとぎ話調のプロローグの雰囲気とちぐはぐになった。
そのため `intro`（夢のシーン。誰かが現れて短く喋る）と `elderVisit`（長老の場面。地の文の説明のみ、
長老の直接セリフは無し）に分割した。長老とちゃんと会話できるのは町で `house-elder` にぶつかったとき。←会話できなくする

## 3. データモデル（`useStoryState.ts`）

`Chapter1State` の全フィールド：

| フィールド | 役割 |
|---|---|
| `scene` | 現在のシーン（2節） |
| `playerPos` | 現在マップ上の座標 |
| `wordsLearned` | 覚えた単語のkana一覧 |
| `bossDefeated` | 章クリア済みか |
| `chestsOpened` | 開けた樽/もらった贈り物のid（二重取得防止） |
| `attackBooks` / `defenseBooks` | ボス戦にだけ自動で乗る持ち越し系ブースト |
| `hpBooks` / `potions` | 持ち物画面から手動で「使う」消費系アイテム |
| `playerHp` / `maxPlayerHp` | 戦闘をまたいで持ち越すHP |
| `journalEntries` | 言霊の書に記録した「旅の記憶」（8節） |

**永続化方針**：`update()` / `learnWord()` / `openChest()` などの操作は、呼ばれるたびに毎回
`localStorage["storyProgress"]` へ即保存する（`saveState`）。つまりゲームは常に自動セーブされている状態。
8節の「言霊の書」に書く「記録する」ボタンは、この自動保存の仕組みとは別の、プレイヤーに見せるための
演出・読み物であって、技術的なセーブ処理ではない（すでに保存済みのものを見せているだけ）。実際の保存処理に変えたい

**バトルとの受け渡し**：`/battle` は別ページなので、`writeStoryBattleResult` /
`readAndClearStoryBattleResult` で `localStorage["storyBattleResult"]` を一度だけ経由してやり取りする
（勝敗・エンカウント種別・HP）。`/story` へ戻るときは Next.js のクライアントキャッシュ再利用を避けるため
`window.location.href` によるハードナビゲーションにしている（`router.push` だとマウント時の効果が
再実行されないことがあった）。

## 4. マップ・当たり判定システム（`chapter1Data.ts`, `types.ts`, `GridExplorer.tsx`）

### 4.1 グリッドと重ね合わせ
マス目自体（`GridMap.tiles`）は `"floor" | "wall"` の2値のみ。井戸・家・NPC・出口・ボスなどは
別配列 `Interactable[]` / `PlacedObject[]` としてマスに重ねる。当たり判定・見た目・会話の3つを
1種類の汎用処理（座標一致判定）にまとめるための設計。

### 4.2 当たり判定は自動計算する（手書きしない）
村の当たり判定は、実際に置いてある家・木・柵・石・花壇・樽などの位置とサイズから
`buildTownTiles()` が自動計算する（`footprintOf` で見た目の矩形を出し、`blockFootprint` で壁にする）。
以前は当たり判定専用のASCII表を手書きしていたが、「物を動かしたのに当たり判定を直し忘れる」
「見た目とズレる」という事故が何度も起きたため、今の自動計算方式に変えた。
物の座標を変えれば当たり判定も自動でついてくる。フィールド（世界地図）の置物はすべて
1x1マスなので、`buildFieldTiles()` では矩形計算をせず該当マスをそのまま壁にするだけでよい。

- `collisionWidthTiles`/`collisionHeightTiles` を見た目のサイズより小さくすると、
  屋根のように「見た目はあるが実際には何もない」部分を壁にせずに残せる（木のてっぺん、家の屋根等）。
- `interactionX`/`interactionY` で、見た目の中心と「ここにぶつかると会話が始まる」座標をずらせる
  （家の壁の中心ではなく、ドアの位置にぶつかり判定を置きたい場合など）。
- 床の**見た目**（草・土の道・水面）は当たり判定と無関係な別レイヤーで、村は`TOWN_FLOOR_ROWS`
  （手書きのASCIIマップ、1文字1マス、48×44）、フィールドは`FIELD_FLOOR_ROWS`（込み入った
  地形がまだ無いので手書きせずコードで生成）とそれぞれ別に管理している。

### 4.2.1 村（`TOWN_MAP`）とフィールド＝世界地図（`FIELD_MAP`）は別マップ
ドラクエのように、村は詳細な1:1スケールのマップ（48×44マス、1マス=`TOWN_TILE_SIZE`=48px）、
フィールドは村の外に広がるもっと大きな縮尺の世界地図（64×48マス、1マス=`FIELD_TILE_SIZE`=80px）
として、それぞれ独立した `GridMap` を持つ。村の門（`town-exit`）にぶつかるとフィールドへ、
フィールドの村アイコン（`field-town-entrance`）にぶつかると村へ、シーンごとテレポートする
（`StoryGame.tsx`の`handleBump`の`kind: "exit"`分岐）。

フィールドが「縮図」らしく見えるように、村より1マスを大きいタイルにして「森」「岩山」の
ようなまとまった地形を表現している（1マス=1つの地形の塊。村のように3x3の細かい木を
並べるのではなく、`FIELD_TREE_SIZE`/`FIELD_STONE_SIZE`はどちらも1x1）。その上でマス数
自体も64×48と村よりだいぶ広くとり、「大陸を歩いているような」歩きごたえのある広さに
してある（最初は12×9の小さいミニマップ→32×24→現在の64×48と、2段階の要望を受けて
拡張してきた経緯がある）。村アイコンから北へ向かって
ボス→未知の領域と奥地が広がる構成で、道中に森・岩山・湖といった地形のかたまりを
点在させ、ただ広いだけでなく探索する意味を持たせている。

村を出るには、必須の言葉（みず・ひ・かぜ）を3つ集めておく必要がある（8.1節）。

### 4.2.2 他の章の本拠地への「行き先」だけ見える目印（`FIELD_LANDMARKS`）
2章「砂漠の町」・3章「妖精の里」は実際に入れるシーンとして`chapter2Data.ts`/
`chapter3Data.ts`に実装済み（`desert-town-entrance`は`kind: "exit"`で実際に入れる。
`magic-forest-view`はまだ「もり」を覚えていないと入れない静的な目印のまま）。
それ以外の未実装の章は、実際に入れるエリアとしては作らず、**世界地図上に見えるだけ・
ぶつかると「まだ早い」と言われるだけの目印**を置くにとどめている。`cave`と同じ
「コトに止められる」演出パターンを流用している：

- `forest-entrance` (2,3)：2章「迷いの森」の入り口。フィールド西側の木立の中
- `snow-mountain-view` (9,2)：3章「氷雪の王国」を遠くに望む地点。フィールド東側の岩山の中

2章以降を実装するときは、`FIELD_LANDMARKS`の該当エントリを本物の入り口
（`kind: "exit"`の新しいシーンへの導線）に差し替えればよい。
「村を出て世界地図を歩いていると、まだ入れない別の場所が視界に入る」という、世界の広がりを
感じさせることが目的なので、中身（実際に歩けるエリア）を伴わない飾りの目印だけを
先に置く、という判断をしている。

### 4.3 徘徊NPC（`useWanderers.ts`）
村人・猫・犬は固定位置ではなく、`spawnX/Y` を中心に `wanderRadius` 内をランダムに歩き回る。
壁・プレイヤー・他の徘徊NPCの現在地を避けて900msごとに1マス移動を試みる。会話中は全員停止する
（`isLocked`）。マップ上は他の物と同じ `Interactable[]` として扱われ、Yソートで自然な前後関係が出る。

### 4.4 カメラ
マップ全体がビューポートより大きい場合、プレイヤーを中心にカメラが追従し、マップ端でクランプされる。

### 4.4.1 Yソートはz-indexで表現する（DOM順で表現しない）
**修正済みのバグ**：以前はスプライト配列を主人公のyで前半・後半に分け、DOM上の並び順
（後に書いた要素が手前に出る）でYソートを表現していた。この方式だと、主人公が1マス
上下に動くたびに「どのスプライトが前半/後半に属するか」が入れ替わり、Reactが該当
スプライトのDOMノードを実際に別の位置へ移動（挿入し直す）必要があった。ノードの移動は
進行中のCSSトランジション（滑らかな移動アニメーション）を中断させてしまうため、
上下移動のときだけカクついて見えていた（左右移動はxしか変わらずYソートの前後関係が
変わらないので、この問題が起きずスムーズだった）。
対策として、DOM上の並び順は固定のまま、CSSの`z-index`だけでYソートを表現する方式に
変えた（`GridExplorer.tsx`）。置物は`sortY * 2`、主人公は`playerPos.y * 2 + 1`という値を
zIndexに使うことで、「sortYが主人公以下の置物は主人公より奥、より大きい置物は主人公より
手前」という以前と同じ見た目を、DOMノードを一切動かさずに実現している。

**この修正が生んだ副作用（修正済み）**：マップが大きい（yが大きい）ほどzIndexの値も
大きくなり、村マップでは80台に達することがある。この値は`GridExplorer`内でのスプライト
同士の前後関係を決めるための「相対値」のつもりだったが、`GridExplorer`のルート要素に
スタッキングコンテキストの区切りが無かったため、`z-index`がページ全体（グローバルな
スタッキングコンテキスト）で評価されてしまい、`StoryDialogue.tsx`の会話ウィンドウ
（`z-50`のオーバーレイ）より木・家のスプライトのzIndexの方が大きくなる場合があった。
その結果、会話中なのに背景の木や家がオーバーレイの上に表示されてしまうことがあった。
対策として、`GridExplorer`のルート要素（`viewportWidth`×`viewportHeight`の枠）に
`isolate`（`isolation: isolate`）を付け、新しいスタッキングコンテキストを作った。
これによりスプライトの`z-index`は常にこの枠の中だけで完結し、外側の会話ウィンドウ等
（`GridExplorer`の兄弟要素）とは値がどれだけ大きくても競合しなくなる。

### 4.5 デバッグ表示
ゲーム中に **g キー**で当たり判定のデバッグ表示（壁=赤、床=緑枠、置物の footprint=水色枠、
ground-levelの物=ピンク枠）を切り替えられる。座標を調整するときはこれを使う。

## 5. 会話・立ち絵システム（`StoryDialogue.tsx`, `DialoguePortrait`）

### 5.1 話者の判定
各行が `"話者「セリフ」"` の形なら話者名とセリフを分離して表示し、その形でない行
（地の文・状況説明）は話者なし扱いにする（`SPEAKER_LINE_PATTERN`）。

### 5.2 立ち絵の左右ルール（`getPortraitsForInteractable`、StoryGame.tsx）
呼び出し側が「誰にぶつかったか」に応じて機械的に立ち絵を決める：
- 樽・贈り物（`grantsItem` あり）：主人公が左、コトが右
- 謎の青年：主人公が左、青年が右
- 猫・犬：コトが左、猫/犬が右（専用の会話用立ち絵が無いのでマップと同じ画像を流用）
- 村人（徘徊NPC）：言葉を教えてくれる相手はコトが左（説明役）、ただ立っているだけの村人は
  主人公が左（コトはまだ出てきていない体）
- 井戸・木・花壇・草：コトのみ（単体で解説する置物）
- 洞窟（`cave`）・森の入り口（`forest-entrance`）・雪山の眺め（`snow-mountain-view`）：
  主人公が左、コトが右（どれもコトが直接止めてくれる演出のため）
- 長老の家（`house-elder`）：主人公が左、コトが右。長老自身のセリフ・立ち絵は
  無し（その場にいない体で、壁越しに喋るのは不自然なため）だが、『言霊の書』
  以外の本が白紙になっていたことをコトと話す場面があるので、コトの立ち絵は出す。
  長老の姿はプロローグの `elderVisit` 場面（専用立ち絵 `tyourou.png`）でのみ見せる
- 村の門（`town-exit`）・フィールドの村アイコン（`field-town-entrance`）：立ち絵無し
  （まだ言葉が足りず止められるときだけ、`handleBump`が直接`[主人公, コト]`を指定する）

### 5.3 マップ画像と会話用立ち絵は同一人物にする
`mura/` のマップ用ドット絵と `kaiwa/` の会話用イラストは別々に生成された素材で、ファイル名の
番号が一致していても同一人物とは限らない（実際、見比べた結果 `kaiwa/murabito1-2` はおばあさん、
`murabito3-4` はひげの職人風、`murabito5-6` は少年、という3人×2パターンだった）。
見た目を比較して手動で対応表（`VILLAGER_PORTRAIT_IMAGES`）を組み、マップで見た人物と会話で
出てくる顔が食い違わないようにしている。専用の会話用素材が無いキャラ（猫・犬）は、マップと
同じ画像をそのまま会話にも使う。

### 5.4 行ごとに変わる背景（`backgrounds` prop）
`StoryDialogue` は `lines` と同じ数（またはそれより少ない数）の画像パスを `backgrounds` に
渡すと、1行ごとに全画面背景を切り替えられる。渡した枚数が `lines` より少ない場合は最後の1枚を
それ以降ずっと表示し続ける仕様にしてあるので、「場面が大きく変わるプロローグは6枚」「キャラが
出てくる前後で2段階あれば十分な場面は2枚」のように、シーンごとに必要な枚数だけ用意すればよい。
画像がまだ無い（ファイルが存在しない）場合は `onError` で読み込み失敗を検知して非表示にするので、
壊れた画像アイコンが出ずに済む＝画像を後から少しずつ足していける。

## 6. 単語（言霊）システム

### 6.1 データ（`CollectibleWord`, `chapter1Data.ts`）
`CHAPTER1_WORD_DICTIONARY` が第1章で手に入る単語の全リスト。`はな`以外の覚えた言葉なら
なんでもボス解放条件にカウントする（ねこ・いぬ等でもOK）方式（`countRequiredWordsLearned`、
`WORD_EXCLUDED_FROM_COUNT`）。`はな`は最初に必ず手に入る言葉なので、これ自体はノーカウントに
して、「はなを覚えたら、そこからさらに3つ集めよう」という導線にしている。
「なまえ」はボス撃破の**報酬**として手に入る単語なので、`WORD_EXCLUDED_FROM_COUNT`（はな）と
同じ理由でカウント対象から除外している（`NAME_UNLOCK_WORD_KANA`。付けると「ボスを倒さないと
手に入らない言葉がボスを倒す条件になる」という矛盾が起きるため）。`countRequiredWordsLearned`
と`WordCollection.tsx`（言葉の図鑑の「★ 外に出る条件にカウントされる」表示）の両方で、
`WORD_EXCLUDED_FROM_COUNT`と`NAME_UNLOCK_WORD_KANA`の両方を除外して表示を揃えている。

### 6.2 単語の入手場所
井戸・家・徘徊NPC・木/花/草の発見スポットなど、「ぶつかると単語を覚える」インタラクションは
すべて `teachesWord` フィールドで表現し、`StoryGame.tsx` の `handleBump` が共通処理する
（`teaches && !alreadyLearned` なら「〜を覚えた！」の行を末尾に追加して `learnWord` を呼ぶ）。

村での発見とは別に、フィールドの戦闘中に敵を倒しても低確率で言葉を見つけられる
（7節の「戦闘中の言葉ドロップ」参照）。村を出る条件（9節）と合わせて、村の中だけを
歩き回るより、思い切って外に出て戦うことにも意味が出るようにしている。

### 6.3 単語の「威力」と難易度カーブ（`battle/difficulty.ts`）
`getWordPower(word) = word.length * (word.length + 5)`。文字数の2乗に比例させることで、
短い単語を連打するより長い単語を使うほうが（打つ時間あたりの実質威力で見て）強くなるように
してある（以前は固定ボーナス付きの式で、短い単語が相対的に強すぎるバランス崩壊があった）。

`pickWordForFloor` は元々試練の塔の「階層が上がるほど長い単語が出やすくなる」重み付け抽選。
ストーリーモードには本来「階層」が無いが、`battle/page.tsx` では単語を出した回数
（`storyTurnCountRef`）を疑似的な階層として渡すことで、同じ仕組みを流用し「戦闘が進むほど
だんだん強い言葉が出る」演出にしている。

### 6.4 敵の言葉とプレイヤーの言葉は別プール
- 自分の攻撃フェーズの単語（`wordList`）＝実際に村で覚えた単語（`CHAPTER1_WORD_DICTIONARY` を
  `wordsLearned` でフィルタ）。何も覚えていない場合だけ `CHAPTER1_FIELD_WORDS` にフォールバック。
- 敵の攻撃（防御フェーズ）の単語（`enemyWordList`）＝ `CHAPTER1_FIELD_WORDS` 固定。
  「敵は必ずしも自分が覚えた言葉ではない、見知らぬ言葉をぶつけてくる」という演出。

`pickNextWord(phase)` / `pickWordForPhase(phase)` が `phase` に応じてどちらのプールを
使うかを切り替える。試練の塔（`!isStory`）はこの区別を持たず、従来どおり1つのプールを共有する。

### 6.5 自分の攻撃を打ち切れなかったときはダメージ0
`resolvePlayerAttack` は `isTimeout`（制限時間内に単語を打ち切れなかった）のとき、
会心・魔法発動を無効化するだけでなく**ダメージそのものを0にする**（`battle/page.tsx`）。
打ち切れなくても多少ダメージが入る仕様だと「焦って早口に打つ」インセンティブが働いて
しまうため、「最後まで正確に打ち切ってはじめて攻撃が成立する」というルールを徹底した。
コンボは従来どおりここで途切れる。ダメージ0のときは「-0」のような表示が出ないよう、
`showDamage` の呼び出し自体をスキップしている。

## 7. 戦闘連携（`battle/page.tsx`, `battle/enemy.ts`）

- `mode=story&chapter=1&encounter=field|boss&words=...&hp=...&maxHp=...&zone=island|mainland`
  （ボスは追加で `attackBooks`/`defenseBooks`。`zone`はフィールド雑魚戦のみ付く）
  というクエリで `/battle` に遷移する。
- `storyEnemies.chapter1`：フィールド雑魚は`regular`（スライムのみ）と`mainlandRegular`
  （スライム+ゴブリン、既存のeasyモード用ゴブリンを流用）の2プールに分かれている
  （7.0.1節）。ボスは「スライムキング」という専用エントリ（画像はスライムを流用した
  プレースホルダー）。
- HPは戦闘をまたいで持ち越す（勝っても負けても村へ戻るときにその時点のHPを引き継ぐ）。
  ただし**負けたときだけ**全回復させてから村へ戻す（0のまま持ち越すと次の戦闘に入った
  瞬間に詰んでしまうため）。
- HPが0になったときの導線は「GAME OVER→村に戻るボタン」という機械的な処理のままだが
  （`battle/page.tsx` 側は変更していない）、`/story` に戻ってきた直後にコトが励ましてくれる
  復活シーン（`REVIVAL_LINES`）を挟むことで、物語の一部として自然に見えるようにしている
  （`StoryGame.tsx` の `readAndClearStoryBattleResult` の `outcome === "lose"` 分岐）。
- 戦闘背景：`STORY_CONFIG.background` は「はじまりの草原」というフィールドの設定に合わせて
  草原の画像を指定している（試練の塔の `easy.png` 等とは別の専用画像）。

### 7.0 フィールドのランダムエンカウント（`StoryGame.tsx`の`handleFieldStep`）
1歩ごとに`ENCOUNTER_CHANCE_PER_STEP`の確率で戦闘に入る（0.15だと敵が多すぎるという
指摘を受け、0.08まで下げた）。さらに、戦闘から戻った直後にまたすぐ次のエンカウントが
発生するとテンポが悪いため、`stepsSinceBattleRef`で歩数を数え、戦闘から戻ってから
`SAFE_STEPS_AFTER_BATTLE`(6)歩は抽選そのものを行わない、という猶予期間を設けている
（`readAndClearStoryBattleResult`の結果処理の先頭でこのrefを0にリセットする）。

### 7.0.1 ゾーン別モンスター出現
フィールドの雑魚戦は、遭遇した場所によって出現する敵が変わる。`handleFieldStep`が
プレイヤーのx座標を`FIELD_MAINLAND_BOUNDARY_X`(113。`FIELD_BRIDGE_TILES`の橋の
x座標と揃えてある)と比較し、`x < 113`（橋を渡った本土側）なら`zone=mainland`、
それ以外（始まりの島側）なら`zone=island`を`/battle`のクエリに付ける。
`battle/page.tsx`はこの`zone`を見て、`storyEnemies.chapter1.regular`（スライムのみ）
と`mainlandRegular`（スライム+ゴブリン）のどちらから雑魚を抽選するか切り替える
（`spawnFloorEnemies`内、`isStory`のときだけ`storyRegularPool`を使う分岐）。
橋が架かる（`bridgeBuilt`）までは本土側に物理的に行けないため、実質的には
「橋を渡った後だけゴブリンが混じるようになる」という後半の変化として機能する。

### 7.1 戦闘中の言葉ドロップ
敵を倒すたびに `WORD_DROP_CHANCE`(0.25) の確率で、まだ覚えていない言葉を1つ見つける
（`tryDropWord`。「なまえ」はボス撃破専用の報酬なのでドロップ対象から除外）。見つけた
言葉は `battleLearnedWordsRef` に貯めておき、勝敗が決まった瞬間（Enterで村に戻るとき）に
`StoryBattleResult.learnedWords` として村側に渡す。**勝敗に関わらず**見つけたものは
持ち帰れる（`StoryGame.tsx` 側で `result.learnedWords?.forEach(learnWord)` を勝敗判定より
先に処理している）。試練の塔の魔法ドロップ（`MAGIC_DROP_CHANCE`）と全く同じ考え方の
言葉版だが、状態を戦闘終了までrefに貯めて一括で持ち帰る点だけが異なる（HP等と違い、
言葉は戦闘中に村側のstateへ直接書き込む手段が無いため）。

## 8. アイテム・経済システム

2種類の性質のアイテムがある：

| 種類 | 入手方法 | 効果が出るタイミング |
|---|---|---|
| 攻撃力の書 / 防御力の書 | 樽 | **持ち越し型**。ボス戦開始時にまとめて自動で倍率が乗る |
| 体力の書 / ポーション | 樽 / 村人 | **消費型**。持ち物画面から「使う」ボタンを押した瞬間に効果 |

- 体力の書：使うとさいだいHPが `ITEM_HP_BONUS` 分上がり、そのままさいだい値まで全回復する
  （試練の塔の宝箱と同じ効果量）。
- ポーション：さいだいHPは変えず、その場でHPを `POTION_HEAL_AMOUNT` 分回復するだけ
  （体力の書より手軽・小型の回復アイテムとして差別化）。
- 樽・贈り物は `chestsOpened` に一度記録されると二度と開けられない（`grantsItem` を持つ
  `Interactable` すべてに共通の仕組み）。
- 内部の関数名・state名（`openChest`, `chestsOpened`）は最初に実装したときの「宝箱」の
  ままにしてあるが、見た目は村の樽・村人からの贈り物に差し替えてある（名前と見た目が
  ズレていることに注意。リネームはまだしていない）。
- ポーションをくれる村人（`wanderer-potion`）は、他の村人と同じ徘徊NPC。専用の見た目を
  用意する代わりに他の村人と同じ画像を使い回している（`WandererDefinition`に`grantsItem`を
  追加拡張し、樽と全く同じ`openChest`の仕組みをそのまま流用できるようにした）。

### 8.1 村を出る条件・洞窟の封印（進行のゲート）

第1章では、必須の言葉（みず・ひ・かぜ）を3つ集めるまでは村の外（フィールド＝世界地図）に
出られない。`town-exit`（村の門、`x:22, y:42`、`kind: "exit"`, `exitsTo: "field"`）に
ぶつかったとき、`handleBump`の`kind === "exit"`分岐が
`countRequiredWordsLearned(wordsLearned) < BOSS_UNLOCK_WORD_COUNT`を判定し、
足りなければコトに呼び止められてシーン切り替え自体が起きない（`return`して終わり）。
3つ集め終わると同じ場所にぶつかってもそのまま`scene: "field"`へ切り替わる。
洞窟（`cave`）も同様に、謎の力に阻まれてコトに止められる形の演出にしてあり、
後の章で入れるようになる伏線として残してある。

次の章以降でボスごとに次のエリアへ進めるようにする場合も、同じ
「`kind: "exit"`のインタラクタブルにぶつかった瞬間に条件をチェックする」パターンを
踏襲するとよい。

### 8.2 名づけイベント（`playerName`, `npcNames`, `NamingModal.tsx`）
- 「はな」を初めて覚えた直後、コトが「そういえば、あなたの名前はなんていうの？」と聞く
  （プレイヤー自身の名づけ。もう決めてある場合は聞き直さない）。
- ボス撃破で「なまえ」を覚える（`ゆうき`に代わる専用報酬。`WORD_DROP_CHANCE`のドロップ
  対象からも除外してある）と、以降`kind:"npc"`の相手（家の住人・徘徊NPC。猫や犬の
  `TOWN_WANDERERS`も同じ`kind:"npc"`なので自動的に対象になる）に初めて話しかけたとき、
  コトが「まだ名前をつけてなかったね。」と聞いてくる（1体につき1回だけ）。
- どちらも同じ`NamingModal`（`allowSkip`は無くし、両方とも常に「名づける/このままにする」
  の2択から始まる）を使う。「名づける」を選んで名前を入力すると、その場で
  `${先頭1文字}、、、、${名前}`という照れ隠しの1行を会話としてはさむ（例：「はな」なら
  「は、、、、はな」）。プレイヤー自身の名前なら主人公が、村人の名前ならその村人自身が
  言った体で表示する（この世界の住人は言葉の力を失って「う、う……」としか喋れない
  設定なので、名前だけは初めて自分の声で言えた、という位置づけ）。「このままにする」を
  選ぶと、`pickRandomName`が`RANDOM_NAME_POOL`（`chapter1Data.ts`）からプレイヤー・村人
  問わずまだ使われていない名前を1つ選ぶ（重複しないように、`playerName`と
  `npcNames`の値をまとめて除外候補にしている）。
- 名づけた村人の名前は、`state.npcNames[id]`に固定され、以降話しかけても聞き直さない。
  会話ウィンドウの話者名（`StoryDialogue`の`title`prop）として出るだけでなく、その村人の
  セリフ本文にある「村人」という文字列（地の文・話者タグ「村人「」」どちらも）を実際の
  名前へ丸ごと置き換える（`TOWN_WANDERERS`のセリフはこの「村人」表記で書かれているので、
  名づけた瞬間に全文が名前入りに変わる。house-elderのようにそもそも「村人」と言わない
  セリフは変化しない）。立ち絵の`name`が`"村人"`のものも同時に名前へ差し替え、
  `StoryDialogue`の話者ハイライト判定がズレないようにしている。
- `playerName`・`npcNames`の値は、タイピングゲームの単語プール（`CHAPTER1_WORD_DICTIONARY`
  経由の`wordList`）には一切混ぜていない。短い名前を選ぶだけで初期から有利になってしまう
  （攻撃力が上がる）事故を避けるため、意図的に表示専用のデータとして分離してある。
- **名づけミッションの完了報酬（`bridgeBuilt`）**：`NAMEABLE_NPC_IDS`（`stranger`を除く
  全`kind:"npc"`）全員分が`state.npcNames`に揃った瞬間、`handleNamingSubmit`が
  `state.bridgeBuilt`を`true`にし、「言霊の力で橋が架かった」という演出を挟む。
  フィールドの地形はちょうど(113,10)/(113,11)が始まりの島↔本土間の幅1マスの海峡に
  なっているので、`bridgeBuilt`のときだけそこを床に変えるだけで橋になる
  （`chapter1Data.ts`の`FIELD_BRIDGE_TILES`/`buildFieldMap`、見た目は`icon_crate_a.png`
  を橋として流用した`FIELD_BRIDGE_OBJECTS`）。`FIELD_MAP`/`FIELD_OBJECTS`という静的な
  export はbridgeBuilt=false固定のまま残してある（`MapEditor.tsx`用）ので、実際の
  ゲーム画面（`StoryGame.tsx`）は必ず`buildFieldMap(state.bridgeBuilt)`/
  `buildFieldObjects(state.bridgeBuilt)`を`useMemo`で呼ぶこと。

## 9. メニューUI（`Menu.tsx`, `KotodamaBook.tsx`, `WordCollection.tsx`, `Inventory.tsx`, `Status.tsx`）

村・フィールド探索中に「📖 メニュー」ボタン1つから開く、タブ切り替え式の画面。
Dialog自体は `Menu.tsx` が1つだけ持ち、4つのタブ内容を出し分ける：

1. **言霊の書**（`KotodamaBook.tsx`）：8節のセーブ演出＋「旅の記憶」の記録一覧
2. **言葉の図鑑**（`WordCollection.tsx`）：6節の単語一覧。覚えた単語は威力も表示、
   ボス解放必須の単語には★マーク
3. **持ち物**（`Inventory.tsx`）：7節の消費/持ち越しアイテム一覧。**0個のものは表示しない**
   （全部0のときだけ「まだ何も持っていない」というプレースホルダーを出す）
4. **ステータス**（`Status.tsx`）：HP、ボス戦ボーナス%、覚えた言葉数、ボス解放条件の一覧

改名し、全部を1つのメニューボタンにまとめた。

## 10. 「言霊の書」＝セーブ演出（`KotodamaBook.tsx`）

主人公は常に『言霊の書』を持っている、という設定にして、「記録する」ボタンを押すと

1. コトが「ここまでの旅を記録しておくね！」と一言
2. 確認バナー（「言霊の書に記録しました。『旅の記憶』が刻まれた。」）を数秒表示
3. `journalEntries` に1行追加（現在は「旅の記憶 その{n}：おぼえた言葉{count}個、HP{hp}/{maxHp}」
   という簡易フォーマット。もっと文脈のある内容にしたければここを膨らませる余地がある）

という演出になる。3節に書いた通り、実際のセーブ処理自体はこのボタンを押さなくても
常に自動で行われている。あくまで「セーブした実感」をプレイヤーに与えるための読み物機能。

## 12. 既知の制約・今後の課題

- 樽・贈り物まわりの内部命名（`openChest`, `chestsOpened`）が見た目（樽・村人）と
  ズレている。実害は無いが、いつかリネームしたい。
- `journalEntries` の中身が簡易的（学習数とHPだけ）。もっとイベント性のある記録
  （「〇〇を覚えた」「ボスに挑んだ」等）にできる余地がある。
- プロローグ以外の背景画像（dream/elder-visit/meet-koto/story-field-battle）は
  ファイル名だけ用意してあり、実画像は順次差し替え中。
- 2・3章は実装済み（上記4.2.2節参照）。4〜8章はまだ`FIELD_LANDMARKS`に「まだ早い」
  プレースホルダーがあるのみで、実装するときは村と同じような専用マップ＋新しいシーンに
  差し替えることになる（着手前にあらためて設計を固めたほうがよい）。
- ポーションをくれる村人は現状1人（`wanderer-potion`）だけ。増やしたい場合は
  `TOWN_WANDERERS`に同じ形で追加すればよい。
- フィールドの地形バリエーションは`FloorTileType`（`types.ts`）を増やすことで対応できる。
  対応する床画像は`public/images/map/okimono/tileset/`に用意し、`GridExplorer.tsx`の
  `FLOOR_TILE_IMAGES`／`mapGen.ts`の`AREA_FLOOR_CHAR_MAP`／`MapEditor.tsx`の床の筆の
  3箇所に登録すると、マップエディタでも塗れるようになる。

### 12.1 第1章を「コトの誘導で進むチュートリアル」にする方針（進行中）
第1章は基本的にチュートリアルにしたい、という方針。まずは言葉を集める・住人に
話しかける、を一通り探索したら村の外へ出られて、次にバトルのチュートリアルをする
ような設計にし、適宜立ち止まって説明を挟む。現状の実装：

- `meetKoto`シーンの最後にコトが「まずは近くにある花壇に行ってみよう」と誘導し、
  シーン終了後は通常の開始位置（井戸のそば）ではなく `TUTORIAL_START_POS`
  （花壇のすぐ南）に降ろすことで、最初の1語（はな）を迷わず取れるようにした。
  花壇の会話に「これが『言葉集め』の基本だよ」という説明セリフも足してある。
- 花壇（`TUTORIAL_TARGET_ID`/`TUTORIAL_TARGET_KANA`＝「はな」）を覚えるまでは、
  `handleBump`の先頭でチェックして、花壇以外の何にぶつかってもその場の会話・
  効果（単語習得、アイテム入手、門の判定など）を処理せず、「コト「あ、まずは
  さっきの花壇に行ってみようよ！」」と軌道修正して返す。寄り道してから花壇に
  戻っても迷わないようにするための安全策（花壇を覚えた瞬間にこの制限は外れ、
  以降は通常どおり自由に探索できる）。
- バトル画面内のチュートリアル演出は実装済み（`battle/page.tsx`の`isTutorial`／
  `tutorialMessage`）。`state.hasHadFirstBattle`が`false`の間だけ最初の1戦に
  `&tutorial=1`が付き、自分の攻撃ターン・相手の攻撃ターンの説明をコトが1回ずつ出す。
  ボス戦・フィールドの雑魚戦のどちらが先でも「最初の戦闘」で出るようにしてあり、
  `town-exit`通過そのものにはもう戦闘トリガーは無い（キングスライムはフィールド上
  (115,10)〜(115,11)に実際に立っている。4.2.2節参照）。
- まだやっていないこと：井戸・家など他の発見スポットへの誘導。
